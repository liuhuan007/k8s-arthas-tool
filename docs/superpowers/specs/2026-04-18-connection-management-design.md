# 连接交互重构设计

- 日期：2026-04-18
- 主题：连接管理从高密度侧栏重构为独立模块
- 状态：已完成设计讨论，待用户审阅

## 1. 背景与问题

当前连接交互存在以下问题：

1. 连接能力放在左侧 sidebar，承担了过多职责。
2. 连接信息展示密度高，集群、命名空间、Pod、连接层级、运行时、状态、操作被压缩在一起。
3. 连接相关操作分散在多个区域：侧栏连接列表、页面内连接按钮、连接状态条。
4. 用户不容易建立稳定心智：
   - 连接在哪里管理
   - 当前连接是谁
   - 当前连接处于 Pod 还是 Arthas 层级
   - 具体功能应该从哪里进入

现有实现中，这种分散是可见的：

- 侧栏连接区：`static/index.html:107`
- 页面内连接按钮：`static/index.html:145`
- 连接状态条：`static/index.html:169`
- 连接列表高密度渲染：`static/js/components/connections.js:127`
- 两步连接状态机：`static/js/components/two-step-connection.js:16`
- AI 助手已经依赖当前连接上下文：`static/js/ai-chat.js:491`

## 2. 设计目标

本次重构目标如下：

1. 把连接管理从高密度侧栏中解耦。
2. 让连接成为一级基础能力模块。
3. 把“连接管理”和“连接内工作”清晰分离。
4. 把系统级页面与连接相关能力彻底分层。
5. 让 AI 助手成为全局可用、连接增强、Arthas 深化的助手入口。
6. 为后续扩展批量管理、权限控制、审计关联和状态解释预留稳定结构。

## 3. 最终信息架构

### 3.1 一级菜单

一级菜单只保留全局、连接无关模块：

- 连接管理
- 模型配置
- 历史记录
- 用户管理
- 审计日志

这些菜单回答的是：系统里有哪些全局模块。

### 3.2 连接管理模块

连接管理作为一级菜单，内部包含两层页面：

#### 连接列表页

用于：

- 查看全部连接
- 筛选连接
- 新建连接
- 清理失效连接
- 进入连接详情

#### 连接详情页

用于：

- 查看单连接信息
- 执行连接管理动作
- 查看连接状态与能力
- 跳转到当前连接下的独立工作页

### 3.3 连接详情下的独立工作页

以下能力不再作为一级菜单，而是作为连接详情下的独立页面入口：

- 🔥 采样工具
- ⚡ Arthas命令
- 🖥️ 终端
- 📊 Pod 监控
- 📂 文件下载
- 🔬 性能诊断

结构变成：

**连接管理 → 连接详情 → 具体工作页**

这层回答的是：针对当前这个连接，我能做哪些事情。

### 3.4 AI 助手定位

AI 助手不放入一级菜单，也不作为连接详情下的普通独立工作页。

AI 助手定位为：

**全局可用、连接增强、Arthas 深化的系统级助手入口**。

它应以顶部入口、固定悬浮入口或右侧抽屉形式存在，并具备以下行为：

- 无连接时：可回答通用问题、引导建连、解释功能入口。
- Pod 连接时：可基于当前连接上下文提供辅助。
- Arthas 连接时：可进入更完整的深度诊断辅助链路。

## 4. 核心设计原则

### 原则 1：连接是工作上下文，不是普通列表项

该项目中的连接不是简单记录项，而是所有诊断能力的上下文入口。因此连接应升级为独立模块，而不是附着在 sidebar 的组件。

### 原则 2：管理和使用分开

- 连接详情页负责“管理”
- 监控、终端、性能诊断、Arthas 命令、采样、文件下载负责“使用”

不把所有能力塞进详情页，避免形成新的超级页面。

### 原则 3：全局菜单只放系统级页面

模型配置、历史记录、用户管理、审计日志等页面与具体连接无关，因此适合作为一级菜单。

### 原则 4：连接相关能力必须挂在连接上下文下

Pod 监控、终端、文件下载、性能诊断、Arthas 命令、采样工具都依赖当前连接上下文，不应与系统级页面并列。

### 原则 5：AI 助手是全局辅助层，不是普通能力页

AI 助手应随处可打开，并感知当前连接与层级，而不是要求用户先进入某个连接页面才能使用。

## 5. 页面设计

### 5.1 连接列表页

#### 页面职责

连接列表页是连接管理模块的主入口，负责管理连接集合。

#### 页面结构

##### 页面头部

左侧：
- 标题：连接管理
- 描述：管理 Pod / Arthas 连接，查看状态并进入详情

右侧：
- 新建连接
- 清理失效连接

##### 筛选区

建议支持：

- 集群
- Namespace
- 连接层级（Pod / Arthas）
- 状态（正常 / 异常 / 失效）
- Pod 名搜索

##### 表格区

推荐字段：

| 字段 | 作用 |
|---|---|
| Pod | 主标识 |
| Namespace | 基础定位 |
| 集群 | 所属集群 |
| 连接层级 | Pod / Arthas |
| 状态 | 正常 / 异常 / 失效 |
| 运行时 | Java / Node / Python |
| 最近检查 | 最后健康检查时间 |
| 操作 | 查看详情 |

##### 交互规则

- 单击整行进入详情。
- 操作列收敛为：查看详情、更多操作。
- 更多操作中包含：健康检查、删除连接。
- 不在表格行直接放“升级 Arthas”按钮，避免列表页重新变成高密度操作区。

### 5.2 连接详情页

#### 页面职责

连接详情页是单连接管理中心，负责回答：

- 这个连接是谁
- 当前状态如何
- 当前是 Pod 还是 Arthas
- 当前可用能力有哪些
- 接下来应进入哪个工作页

#### 页面结构

##### A. 基本信息区

展示稳定信息：

- Pod 名
- Namespace
- 集群
- Container
- 创建时间

##### B. 当前状态区

展示动态状态：

- 当前层级：未连接 / Pod / Arthas
- 当前状态：正常 / 异常 / 失效
- 最近检查时间
- 当前异常说明

##### C. 运行时与能力区

拆成两个层次：

###### 运行时信息
- runtime 类型
- 版本
- Java PID（如有）

###### 能力信息
- 当前已解锁能力
- 升级后可解锁能力

视觉上必须明确区分连接层级和功能能力，不能混在同一层里。

##### D. 操作区

根据状态展示不同主动作：

###### 未连接
- 建立 Pod 连接

###### Pod 已连接 + Java
- 升级到 Arthas
- 健康检查
- 删除连接

###### Pod 已连接 + 非 Java
- 健康检查
- 删除连接

###### Arthas 已连接
- 健康检查
- 重新连接
- 删除连接

##### E. 能力入口区

连接详情页底部或侧边提供当前连接的工作入口：

**基础操作**
- 🖥️ 终端
- 📊 Pod 监控
- 📂 文件下载

**诊断分析**
- 🔬 性能诊断
- ⚡ Arthas命令
- 🔥 采样工具

这些入口都带着当前连接上下文跳转到独立页面。

### 5.3 独立工作页

这些页面只承载实际工作流，不再承担连接管理本身：

- Pod 监控页
- 终端页
- 文件下载页
- 性能诊断页
- Arthas 命令页
- 采样工具页

每个页面顶部建议统一显示：

- 当前连接名称
- 当前层级
- 返回连接详情

这样工作页不会丢失上下文。

### 5.4 全局当前连接上下文条

#### 职责

只做轻量上下文展示，不做重管理。

#### 展示内容

- 当前连接：cluster / namespace / pod
- 当前层级：未连接 / Pod / Arthas
- runtime 摘要
- 查看详情按钮

#### 不承担的职责

- 不做批量操作
- 不做升级主入口
- 不做删除
- 不承载复杂状态解释

## 6. 典型用户路径

### 路径 1：首次建立连接

1. 进入连接管理
2. 点击新建连接
3. 选择 cluster / namespace / pod / container
4. 建立 Pod 连接
5. 若支持，则升级 Arthas
6. 进入连接详情
7. 从详情进入监控、诊断、终端等页面

### 路径 2：已有连接继续使用

1. 进入连接管理列表
2. 找到目标连接
3. 查看状态
4. 进入连接详情
5. 执行检查、升级、删除或跳转工作页

### 路径 3：从工作页回到连接管理

1. 用户在监控、诊断、终端等工作页使用功能
2. 顶部展示当前连接上下文
3. 点击“查看连接详情”
4. 返回对应连接详情页

### 路径 4：通过 AI 助手获取帮助

1. 用户在任意页面打开 AI 助手
2. AI 助手感知当前连接上下文
3. 无连接时引导建连或解释功能入口
4. Pod 连接时提供上下文感知帮助
5. Arthas 连接时提供更深入的分析建议

## 7. 与现有代码的映射建议

### 7.1 `static/js/components/connections.js`

建议保留为：

- 连接数据模型层
- 连接状态访问层
- 当前连接同步层

适合继续复用的能力：

- `getConnections()`
- `getCurrentConnection()`
- `addConnection()`
- `removeConnection()`
- `inferConnLevel()`
- `getConnRuntime()`
- `canUpgradeConnection()`

建议调整：

- `renderConnList()` 从 sidebar 风格重构为表格视图渲染。

### 7.2 `static/js/components/two-step-connection.js`

建议沉淀为：

- 连接详情页状态机
- Pod 连接 / Arthas 升级流程控制器

适合继续复用：

- `ConnectionState`
- `podConnect()`
- `upgradeToArthas()`
- 运行时与能力展示相关逻辑

### 7.3 `static/js/app-ui.js`

建议逐步收缩其职责，只保留：

- 初始化协调
- 全局事件分发
- 当前连接上下文同步

不再承担连接主界面渲染中心角色。

### 7.4 `static/index.html`

当前：

- sidebar 中有连接列表
- 主区有连接按钮
- 页面中有状态条

建议重构方向：

- sidebar / 菜单只保留全局导航
- 连接管理成为独立页面
- 状态条保留为“当前连接上下文条”
- 连接动作集中到连接详情页

### 7.5 `static/js/ai-chat.js`

该模块已经具备“全局入口、感知当前连接”的基础：

- 监听连接切换：`connection-changed`
- 优先使用当前活跃连接
- 同步本地连接状态

建议未来产品结构层面延续这一方向：AI 作为全局助手入口存在，而不是普通连接内页面。

## 8. 为什么这个方案优于当前结构

### 当前结构的问题

- 左侧承担了过多连接信息和操作。
- 连接入口分散。
- 状态与操作混杂。
- 用户不清楚连接管理的主入口在哪里。
- 连接相关能力与系统级页面边界不清晰。

### 新结构的优势

- 结构清晰：列表 → 详情 → 工作页。
- 菜单层级明确：全局页面 vs 连接上下文能力。
- 连接心智稳定：先找连接，再进入工作。
- AI 助手定位清晰：全局可用但感知连接。
- 扩展性强：后续加批量操作、权限控制、审计关联都更自然。

## 9. 最终推荐结论

推荐采用以下架构作为重构目标：

### 一级菜单
- 连接管理
- 模型配置
- 历史记录
- 用户管理
- 审计日志

### 连接管理模块
- 连接列表页（表格）
- 连接详情页

### 连接详情页中的能力入口
- 🔥 采样工具
- ⚡ Arthas命令
- 🖥️ 终端
- 📊 Pod 监控
- 📂 文件下载
- 🔬 性能诊断

### 全局入口
- 🤖 AI 助手（全局入口，不占一级菜单）

该结构兼顾清晰度、可扩展性和产品心智稳定性，适合作为本次连接交互重构的设计基线。
