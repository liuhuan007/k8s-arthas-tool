## 4. 新增问题与建议

### 4.1 🔴 关键问题：Tunnel Server 与 Port-Forward 的架构冲突

**文档位置**: `05-toolbox.md` 第3节

**问题描述**:

文档同时描述了两种连接模式：
1. **Port-Forward 模式**（当前实现）：通过 `kubectl port-forward` 建立本地到Pod的隧道
2. **Tunnel Server 模式**（新增设计）：Pod 主动连接到 Tunnel Server，平台通过 Tunnel Server 访问

这两种模式在架构上存在冲突：
- Port-Forward 需要平台主动建立连接，Pod 被动监听
- Tunnel Server 需要 Pod 主动注册，平台被动接收

**风险**:
- 用户可能混淆两种模式的适用场景
- 连接管理代码需要同时支持两种模式，增加复杂度
- 状态机需要处理两种不同的连接生命周期

**建议**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    连接模式决策树                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  是否有网络限制？                                                │
│       │                                                         │
│       ├── 否 ──→ 使用 Port-Forward（简单直接）                  │
│       │                                                         │
│       └── 是 ──→ 是否有 Tunnel Server？                         │
│                    │                                            │
│                    ├── 否 ──→ 提示部署 Tunnel Server            │
│                    │                                            │
│                    └── 是 ──→ 使用 Tunnel Server                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**建议在文档中明确**：
1. 两种模式的适用场景和优先级
2. 状态机如何统一处理两种模式
3. 连接管理器的抽象层设计

### 4.2 🔴 关键问题：Skill DSL 执行引擎的幂等性

**文档位置**: `03-diagnosis-center.md` 第5节、`12-skill-registry-orchestrator-gateway.md`

**问题描述**:

Skill DSL 支持多步骤执行，但未明确：
1. 如果执行中途失败，已执行的步骤是否需要回滚？
2. 重试时，是重试整个 Skill 还是只重试失败的步骤？
3. 某些步骤（如 `redefine`）有副作用，如何保证幂等性？

**示例场景**:

```yaml
steps:
  - step_type: arthas_command
    command: "jad --source-only com.example.Service"
  - step_type: arthas_command
    command: "mc /tmp/Service.java"
  - step_type: arthas_command
    command: "redefine /tmp/Service.class"
```

如果步骤 2 失败，步骤 1 已经执行（jad 查看），用户重试时：
- 是否重新执行步骤 1？
- 步骤 2 生成的 class 文件是否还存在？

**建议**:

1. **步骤状态持久化**：每个步骤的执行状态写入 `step_logs`
2. **重试策略配置**：Skill 定义中增加 `retry_policy` 字段

```yaml
retry_policy:
  mode: "resume"  # resume: 从失败步骤继续 | restart: 重新开始
  max_retries: 3
  retryable_steps: [1, 2]  # 可重试的步骤
  non_retryable_steps: [3]  # 不可重试的步骤（有副作用）
```

3. **幂等性标记**：步骤类型增加 `idempotent` 属性

| step_type | 幂等性 | 重试建议 |
|-----------|--------|---------|
| `arthas_command` (只读) | ✅ 幂等 | 安全重试 |
| `arthas_command` (有副作用) | ❌ 非幂等 | 需要确认 |
| `llm_analysis` | ✅ 幂等 | 安全重试 |
| `get_pod_status` | ✅ 幂等 | 安全重试 |

```sql
CREATE TABLE agent_sessions (

```

3. **连接绑定**：会话与连接强绑定，连接断开时会话标记为 `suspended`

### 4.4 🟡 中等问题：诊断报告的存储与检索

**文档位置**: `08-frontend-design-detail.md` 第1.3节

**问题描述**:

前端设计了详细的诊断报告页面，但后端未明确：
1. 报告数据从哪里读取？（`task_logs`？独立的 `reports` 表？）
2. 报告是否支持导出为 PDF？
3. 报告是否支持分享链接？

**建议**:

1. **报告数据来源**：

```python
def get_diagnosis_report(run_id: str) -> dict:
    """获取诊断报告"""
    run = db.fetch_one("SELECT * FROM task_logs WHERE id = ?", (run_id,))
    steps = db.fetch_all("SELECT * FROM step_logs WHERE run_id = ? ORDER BY step_num", (run_id,))
    
    return {
        "run": run,
        "steps": steps,
        "summary": run["result_json"],  # AI 生成的摘要
        "connection_snapshot": json.loads(run["connection_snapshot_json"]),
        "capability_snapshot": json.loads(run["capability_snapshot_json"])
    }
```

2. **报告导出**（P1）：

2. **请求重试**：

```javascript
async function fetchWithRetry(url, options, maxRetries = 3) {
    for (let i = 0; i < maxRetries; i++) {
        try {
            const response = await fetch(url, options);
            if (response.ok) return response;
        } catch (error) {
            if (i === maxRetries - 1) throw error;
            await new Promise(r => setTimeout(r, 1000 * (i + 1)));  // 指数退避
        }
    }
}
```

3. **防抖节流**：

```javascript
function debounce(fn, delay) {
    let timer;
    return function (...args) {
        clearTimeout(timer);
        timer = setTimeout(() => fn.apply(this, args), delay);
    };
}

// 使用示例
const debouncedSearch = debounce((query) => {
    connectionSelector.filterConnections(query);
}, 300);
```

## 5. 架构决策记录

### ADR-001：任务中心不直接拥有连接执行能力

- **决策**：任务中心负责 run 生命周期、日志、调度和产物；具体 Pod/Arthas 操作委托现有执行器。
- **原因**：避免重复实现 kubectl、port-forward 和 Arthas HTTP 调用，减少竞态。
- **代价**：需要定义清晰的执行器接口和连接快照结构。

### ADR-002：即时诊断复用运行记录，不创建任务定义

- **决策**：即时诊断创建 `task_logs`，定时任务创建 `task_definitions + task_logs`。
- **原因**：符合线上排障即时性，同时保留定时任务可复用配置。
- **代价**：历史查询需要同时支持"有 task_definition"和"无 task_definition"的 run。

### ADR-003：前端当前连接与运行中任务解耦

- **决策**：连接切换只影响后续新建诊断，不自动取消已运行诊断。
- **原因**：多连接排障是高频场景，运行任务应该以服务端 `run_id` 为准。
- **代价**：前端需要运行中任务列表和连接快照展示。

### ADR-004：连接模式统一抽象

- **决策**：定义 `ConnectionProvider` 接口，统一 Port-Forward 和 Tunnel Server 两种模式
- **原因**：避免连接管理代码分裂，支持未来扩展新连接模式
- **代价**：需要抽象层代码，增加初期开发成本

```python
from abc import ABC, abstractmethod

class ConnectionProvider(ABC):
    """连接提供者抽象接口"""
    
    @abstractmethod
    async def connect(self, pod_target: PodTarget) -> Connection:
        """建立连接"""
        pass
    
    @abstractmethod
    async def disconnect(self, connection_id: str) -> bool:
        """断开连接"""
        pass
    
    @abstractmethod
    async def health_check(self, connection_id: str) -> HealthStatus:
        """健康检查"""
        pass

class PortForwardProvider(ConnectionProvider):
    """Port-Forward 连接提供者"""
    pass

class TunnelServerProvider(ConnectionProvider):
    """Tunnel Server 连接提供者"""
    pass
```

### ADR-005：诊断报告与执行记录分离

- **决策**：诊断报告从 `task_logs` 动态生成，不单独存储
- **原因**：避免数据冗余，保证报告与执行记录一致性
- **代价**：报告生成需要查询多表，可能有性能开销

### ADR-006：监控数据分层存储

- **决策**：P0 使用内存存储，P1 迁移到 SQLite，P2 考虑时序数据库
- **原因**：渐进式实现，避免过早优化
- **代价**：需要设计统一的监控数据访问接口
