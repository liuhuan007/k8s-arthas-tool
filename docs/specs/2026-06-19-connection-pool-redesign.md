# 连接池重构设计规格 v2

## [S1] 问题

当前连接模型是"单连接"设计：只能有一个活跃连接，切换 Pod = 断开旧连接 + 重连新 Pod。用户需要在多集群多 Pod 间快速切换诊断，当前模型不满足需求。

### 1.1 根因
- `_currentConnId` 只能指向一个 Pod
- 切换触发断连/重连，耗时且不稳定
- 状态碎片化（`_connState`、`_connections`、`ConnectionStore` 散落各处）
- 切换后 UI 卡死（按钮禁用、面板不更新）

### 1.2 影响
- 无法快速在多个 Pod 间对比诊断
- 切换耗时 2-5 秒（需要重建 port-forward 和 Arthas 连接）
- 连接状态不一致导致 UI bug

## [S2] 解决方案：连接即工作区

**核心模型变更：** 从"单连接"到"连接池 + 独立工作区"

```
连接池（多个并存）  →  选中一个  →  该 Pod 的完整工作区
                                     ├── 监控（per-connection）
                                     ├── 采样（per-connection）
                                     ├── Arthas（per-connection）
                                     ├── 终端（per-connection）
                                     ├── 文件（per-connection）
                                     └── 历史（per-connection）
```

### 2.1 连接池
- 多个连接并存，每个连接独立维护
- 切换焦点 = 切换视图，不触发网络操作
- 已建立的连接在后台保持（port-forward、Arthas session 不中断）
- **每个连接独立 port-forward**：每连接一个进程，切焦点不关旧连接

### 2.2 工作区（per-connection）
- 每个连接有独立的 Tab 栏
- Tab 内容基于连接层级动态生成
- **Tab 状态完整保留**：切换到 Pod-B 后，Pod-A 的 Tab 状态（当前 Tab、子 Tab、滚动位置）完整保留
- **数据刷新策略**：切回时自动获取最新数据 + 保持上次数据 + 手动刷新按钮

## [S3] 连接生命周期

### 3.1 状态定义

```
State            说明                    触发
──────────────── ─────────────────────── ──────────────────
DISCONNECTED     未连接                  初始 / 手动断开
CONNECTING       连接中                  点击"连接"
POD_CONNECTED    Pod 已连接              kubectl exec 通道建立
ARTHAS_UPGRADING Arthas 启动中          点击"启动 Arthas"
ARTHAS_READY     Arthas 就绪             Arthas HTTP 可达
DEGRADED         连接弱/响应慢           心跳超时 > 3 次
DEAD             连接失效                心跳超时 > 10 次
```

### 3.2 心跳机制
- **检测方式**：两者结合
  - kubectl get pod 检查 Pod 存在性
  - kubectl exec echo 检查连接可用性
- **频率**：每 5 秒一次轻量心跳
- **容错**：连续 3 次失败 → DEGRADED，连续 10 次失败 → DEAD
- **恢复**：DEGRADED → POD_CONNECTED（自动），DEAD 需手动重连

### 3.3 自动重连
- **默认关闭**（多 Pod 场景避免抢占资源）
- 可选 per-connection 开关
- 开启后：DEAD 30 秒后自动尝试重连
- 重连失败 3 次后停止，通知用户

### 3.4 页面刷新恢复
- **配置持久化 + 运行时重建**：连接配置存 localStorage，运行时状态（健康/层级）每次从后端重建
- 刷新后自动恢复所有之前活跃的连接
- 焦点连接自动恢复

### 4.2 连接卡片

**默认简洁模式**（一行）：
```
[🟢] user-service-7b8f  [Arthas]
     prod-1 / app-prod  java
```

**展开详情**（点击⋯按钮）：
```
[🟢] user-service-7b8f  [Arthas]
     prod-1 / app-prod  java
  ┌─────────────────────────────┐
  │ Pod: user-service-7b8f...   │
  │ 运行时: java 17.0.2         │
  │ PID: 12345 · 运行: 72h     │
  │ Arthas: v3.7.1 :8563       │
  │ ✅ 健康 · 心跳: 2s前        │
  │ ☑ 自动重连                  │
  │ [🚀 Arthas][⏹ Arthas]       │
  │ [🔌 断开][🔄 重启][✕ 删除]  │
  └─────────────────────────────┘
```

支持**紧凑/详细两种视图切换**

### 4.3 连接分组

按状态分组，活跃连接默认展开，已断开默认折叠：
```
活跃连接 (2)
  🟢 user-service-7b8f  Arthas
  🟢 order-service-5d4a Pod
已断开 (1)
  🟡 nginx-proxy-1a2b   ⚠弱
```

**排序规则**：最近使用优先

### 4.4 连接池搜索
- 文本搜索：Pod 名、集群名、命名空间
- 实时过滤

### 4.5 连接池为空时的引导
- 连接池为空时显示引导卡片
- 提示"添加第一个连接"，引导用户点击"+ 新连接"

### 4.6 连接状态指示

| 指示器 | 颜色 | 说明 |
|--------|------|------|
| 🟢 绿色脉冲 | `#34C759` | Arthas Ready |
| 🔵 蓝色 | `#007AFF` | Pod Connected |
| 🟡 黄色闪烁 | `#FF9500` | CONNECTING / DEGRADED |
| 🔴 红色 | `#FF3B30` | DEAD |
| ⚫ 灰色 | `#344060` | DISCONNECTED |

### 4.7 工作区 Tab 栏

Tab 按连接层级动态生成：

**Pod 层级可用：**
- 📊 监控（概览/指标/进程/网络/磁盘/事件/日志/配置）
- 🖥️ 终端
- 📂 文件
- 📋 历史

**Arthas 层级额外可用：**
- 🔥 采样（CPU/内存/锁/Wall/JFR/线程Dump/HeapDump）
- ⚡ Arthas 命令
- 🔧 热修复
- 🔬 诊断

## [S5] 连接池操作

### 5.1 添加连接
- **手动**：点击"+ 新连接" → 选集群/命名空间/Pod → 连接
- **自动发现**：
  - Namespace 级扫描：选中 namespace 后列出所有 Pod
  - Deployment 级扫描：可选展开到 Deployment 级
  - 支持两种粒度

### 5.2 连接详情位置
- 点击连接卡片后，**右侧工作区上方**显示连接详情面板
- 详情包含：集群/命名空间/Pod名/运行时/PID/运行时间/Arthas版本
- 快捷操作按钮在详情面板中

### 5.3 删除连接
- 弹窗确认
- 提示"采样数据保留在历史记录中"
- 删除连接记录，不影响文件系统中的采样文件

### 5.4 断开连接
- 弹窗确认，提示"采样数据不删除"
- 关闭 port-forward 和 Arthas session
- 连接状态变为 DISCONNECTED
- 不影响其他连接

### 5.5 连接数量上限
- 不限制，建议不超过 10 个活跃连接

## [S6] 数据隔离

### 6.1 跟随连接切换（per-connection）
- 监控数据（CPU/内存/网络/磁盘/事件/日志/配置）
- 采样任务和结果
- Arthas 会话和命令历史
- 热修复工作流
- 终端会话
- 文件浏览器
- 历史记录（按 Pod 过滤）

### 6.2 全局共享
- 连接池本身
- 模型配置 / MCP 接入
- 任务中心
- 工具箱
- 用户管理 / 审计日志

### 6.3 断开/删除时的数据保留
```
断开连接时：
  ✅ 关闭 port-forward 进程
  ✅ 标记状态为 DISCONNECTED
  ✅ 释放 Arthas session
  ✅ 停止心跳/监控轮询
  ❌ 不删除 profiler_output 中的采样文件
  ❌ 不删除 profiler_tasks 数据库记录
  ❌ 不删除诊断历史
  ❌ 不影响其他连接的采样数据
```

### 6.4 采样输出路径
保留现有路径：`data/profiler/{cluster}/{namespace}/{appName}/`

## [S7] 迁移策略

### 一次性替换
- 一次性重写所有连接相关代码
- ConnectionStore 重构为多连接存储
- 旧的 `_currentConnId` / `_connState` 等全局变量全部删除
- 旧的 Pod 目标区替换为连接池

### 实施阶段
Phase 1: 连接池存储层 → Phase 2: 连接生命周期 → Phase 3: UI 重构 → Phase 4: 功能迁移 → Phase 5: 数据层适配
